# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

`tlschecker` is an experimental TLS/SSL certificate checker. It connects to remote hosts, inspects the X.509 certificates they present, and reports expiry, chain, revocation status, and a configuration grade. It ships as **both a library and a binary from one package** (`[[bin]]` + `lib.rs`).

## Commands

```sh
cargo build                      # debug build (openssl is vendored — needs a C toolchain + perl, no system openssl)
cargo build --release            # release build
cargo clippy --verbose           # lint (run in CI; keep clippy-clean)
cargo test                       # offline tests only
cargo test -- --ignored          # network integration tests (connect to google.com, *.badssl.com, digicert.com)
cargo test <name>                # run a single test, e.g. cargo test test_parse_host_port_with_port
cargo test <name> -- --ignored   # run a single network test
cargo run -- example.com         # run the CLI locally
```

MSRV is **1.88.0** (`rust-version` in `Cargo.toml`, required by ratatui 0.30); CI tests against both MSRV and stable. The binary requires the default-on `tui` cargo feature (`[[bin]] required-features`); library consumers can use `default-features = false` to skip the ratatui stack.

### Test convention (important)

Tests are split by network dependency:
- **Offline tests** run on every `cargo test`. Cert-parsing logic is tested with **synthetic X.509 certs generated in-process** (`make_test_x509`, `make_test_x509_signed_by`, `make_test_x509_weak_sig` in `lib.rs` tests) and with hand-built `TLS` structs (`make_test_tls`).
- **Network tests are marked `#[ignore]`** and only run with `-- --ignored`. They hit real external hosts and tolerate flaky responders (they accept `Good`/`Unknown` for revocation rather than asserting one).

When adding logic that touches a live TLS connection or OCSP/CRL responder, prefer an offline test using synthetic certs; gate anything that needs the network behind `#[ignore]`.

## Architecture

**`src/lib.rs` (`tlschecker` crate) — the core.** Everything funnels through `TLS::from(host, port, check_revocation, calculate_grade)`: it does TCP connect → TLS handshake → certificate/cipher extraction → optional revocation → chain analysis → hostname validation → optional grading, returning a `TLS` struct (`Cipher` + `CertificateInfo` + optional `TLSGrade` + optional `TlsScan`). All public types (`CertificateInfo`, `Subject`, `Issuer`, `Chain`, `RevocationStatus`, `SecurityWarning`, `TLSError`) and the free functions (`check_revocation_status`, `check_ocsp_status`, `check_crl_status`, `find_issuer_cert`, `analyze_certificate_chain`, `cert_matches_hostname`, `is_self_signed_certificate`) live here.

> **Design decision — do not "fix":** the handshake uses `set_verify(SslVerifyMode::empty())` (`TLS::from`). Peer verification is intentionally disabled because this is a diagnostic tool that must inspect expired, self-signed, and otherwise-invalid certificates. Trust decisions are made *after* inspection (self-signed detection, chain analysis, hostname matching, revocation), not by the TLS stack.

**Revocation flow:** `check_revocation_status` tries OCSP first (`check_ocsp_status`), falling back to CRL (`check_crl_status`) when OCSP is inconclusive. Both fetch URLs embedded in the certificate (AIA OCSP responders / CRL distribution points). OCSP **and** CRL responses have their signatures verified against the issuer before any status is trusted; verification failure degrades to `Unknown` (which triggers the CRL fallback) rather than a trusted result.

**Certificate analysis & warnings:** `analyze_certificate_chain` produces `SecurityWarning`s for weak signature algorithms, incomplete chains, out-of-order chains (`is_chain_well_ordered`), and expiring/expired intermediates (within `CHAIN_EXPIRY_WARNING_DAYS` = 30; the leaf and self-signed roots are skipped). Hostname validation is separate: `cert_matches_hostname` checks SAN DNS entries (with single-label `*.` wildcard support, `matches_dns_name`) and falls back to the CN only when there are no DNS SANs; a mismatch is appended as `SecurityWarning::HostnameMismatch` in `TLS::from`. Each warning category that affects trust is mirrored into `GradingInput` (e.g. `has_hostname_mismatch` caps the grade at C, like self-signed). When adding a new `SecurityWarning` variant, update both formatters' match arms in `main.rs` and the `GradingInput` wiring if it affects trust. `CertificateInfo` also carries `cert_sha256`/`cert_sha1` fingerprints (colon-hex, computed by `fingerprint()`) and a `#[serde(skip)]` `pem` field (full presented chain, leaf first) used by `--export-pem`. `cert_sn` is colon-hex (via `serial_hex`), not the decimal value of the ASN.1 INTEGER, so it matches CA portals, browsers, and `openssl x509 -text`.

**Issuer/revocation URLs.** `ocsp_urls` / `ca_issuer_urls` / `crl_urls` expose the endpoints embedded in the certificate. `ocsp_urls` comes from `openssl`'s `ocsp_responders()` — the *same* accessor `check_ocsp_status` queries, deliberately, so displayed responders can't drift from queried ones; `crl_urls` comes from the shared `crl_urls()` helper that `check_crl_status` also iterates (it collects every URI across all distribution points, so all are tried). `openssl` has no caIssuers getter, so `certext::ca_issuer_urls` walks the AIA extension (`1.3.6.1.5.5.7.1.1`) with the `der` primitives, filtering on the id-ad-caIssuers access method — same best-effort, never-panics contract as the rest of `certext`.

**`src/probe.rs` — protocol/cipher enumeration (`--scan`).** `scan_tls(host, port)` returns a `TlsScan` by attempting many single-version handshakes (`try_handshake` pins min==max proto version and one cipher / TLS 1.3 ciphersuite at a time) across SSLv3–TLS 1.3, enumerating accepted ciphers from curated candidate lists. It is opt-in and slow (many short connections). Note `scan_tls` only yields useful data for hosts reachable at ≥TLS 1.2, because `TLS::from` must complete a handshake first (modern OpenSSL's default floor is TLS 1.2) and the scan only runs in the `Ok` branch. Each `ProtocolSupport.version` is a typed `ProtoVersion` enum (not a string); `ProtoVersion::label()` is the single source for its display/JSON form, so `analyze_scan`/`scan_supports_obsolete_protocol` match on variants and a label change can't silently desync from the detection logic.

`--scan` implies `--grade` (`main.rs::should_grade` = `config_grade || scan`): a scan is only surfaced in the summary table through the grade, so grading is force-enabled whenever scanning. `TLS::from` then computes a negotiated-only grade and `apply_scan` upgrades it.

**Folding scan into results (`TLS::apply_scan`).** The `main.rs` worker calls `cert.apply_scan(scan)` after `TLS::from` (keeping `TLS::from`'s signature stable). `apply_scan` runs `analyze_scan()` — which emits `WeakProtocol` (SSLv3/TLS 1.0/1.1) and `WeakCipher` (`is_weak_cipher`: RC4/DES/NULL/EXPORT/anon/MD5) warnings — appends them to `security_warnings`, then recomputes the grade if one was present. Grade inputs are built in one place, `build_grading_input(cipher, certificate, Option<&TlsScan>)`, used by both `TLS::from` (scan `None`) and `apply_scan` (scan `Some`); it derives the trust flags from `security_warnings`, `supports_obsolete_protocol` (SSLv3/TLS 1.0) from the scan, and `accepts_weak_cipher` from **both** the negotiated cipher name (`is_weak_cipher`, so a weak negotiated cipher caps the grade even without `--scan`) and the scan's accepted-cipher list. Both flags cap the composite grade at 69 (C max), alongside the existing self-signed / hostname-mismatch caps.

**`src/ct.rs` — Certificate Transparency lookup (`--ct-check`).** `check_ct_status(cert_sha256)` queries [crt.sh](https://crt.sh) by the leaf's SHA-256 fingerprint (an exact per-cert lookup that avoids the result-set caps a domain query hits on busy hosts). crt.sh has no JSON for fingerprint queries, so `parse_ct_html` matches the two stable HTML signals — a `?id=<n>` detail link (logged) or `Certificate not found` (absent). `CtStatus` is a **tri-state enum** mirroring `RevocationStatus`: `Logged { crtsh_id, crtsh_url }` / `NotLogged` / `Unknown`. `check_ct_status` returns the two *definitive* outcomes as `Ok` and "could not check" (network failure / unrecognized response) as `Err`; the worker maps that `Err` to `CtStatus::Unknown` and logs the reason via `warn!` (stderr) so a crt.sh outage is never collapsed into `NotLogged`. Like `--scan`, it is opt-in and network-bound, a **pure CLI flag off `Args`** (not part of `Config`), and folded in after `TLS::from` via `TLS::apply_ct(ct)` (mirroring `apply_scan`): only `NotLogged` appends `SecurityWarning::NotInCertificateTransparency` (`Unknown` adds no warning), then stores the status on `TLS.ct` (`Some` ⇔ `--ct-check` was requested). CT inclusion is **informational only** — deliberately *not* wired into `GradingInput`/grade caps, because legitimately private certs are often absent from public CT logs. Output: TextFormat prints a "Certificate Transparency:" block (yes/no/unknown); SummaryFormat shows a conditional `CT` column (`✓`/`✗`/`?`, hidden unless some host was checked — same pattern as the Grade column); JSON serializes `TLS.ct`; both formatters' warning match arms handle the new variant. Note: the tracing subscriber writes to **stderr** (`with_writer(std::io::stderr)`) so diagnostics never corrupt the stdout JSON/text stream.

**`src/sct.rs` — offline embedded-SCT parsing.** `embedded_scts(cert)` reads the leaf's Signed Certificate Timestamps from the X.509 extension OID `1.3.6.1.4.1.11129.2.4.2` (RFC 6962 §3.3) — offline proof the cert was *submitted* to CT logs, complementing the network `ct` *inclusion* check. The `openssl` crate has no generic extension accessor, so it is **dependency-free**: a small DER TLV walker (`read_tlv` → `find_extension_value`) structurally descends Certificate → tbsCertificate → the `[3]` extensions wrapper → each Extension (not a byte-pattern scan), then `parse_sct_list` decodes the TLS-serialized SCT list and `format_timestamp_ms`/`civil_from_days` render timestamps without a date crate. It is **best-effort and never errors** — absent/malformed structures yield an empty `Vec<Sct>`. Computed **always** (offline, cheap — no flag) in `get_certificate_info`, stored on `CertificateInfo.scts` (`#[serde(default, skip_serializing_if = "Vec::is_empty")]`). TextFormat prints an "Embedded SCTs" block when present and cross-references it in the `--ct-check` `Unknown` branch ("certificate was submitted to CT logs"); summary is unchanged (SCTs are always-on, so they don't drive the opt-in CT column).
**`src/grading.rs` — deterministic A+→F grading.** Pure functions over a transient `GradingInput`. Weighted composite: Protocol 30% · Cipher 25% · Key Exchange 15% · Cert Key Size 15% · Trust 15%, then **hard caps** (cap values are the tops of the `score_to_letter` bands, so they mean "this letter at best"): expired/revoked → 0 (F); negotiated TLS 1.0 or SSLv3 (protocol score ≤ 20) → cap 54 (D max); self-signed / hostname mismatch / weak cipher / scan-discovered obsolete protocol → cap 69 (C max). Grade is computed only when `--grade` is set and is serialized into `TLS` (skipped when `None`).

**`src/main.rs` — the CLI.** clap arg parsing, then either the interactive dashboard (see `src/tui/`) or output via the `Formatter` trait (`JsonFormat` / `TextFormat` / `SummaryFormat`) selected through `FormatterFactory`. **Output mode rule (`use_dashboard`):** the dashboard is the default when stdout is a TTY; piped/redirected stdout, an explicit `-o`, a config-file `output` key, or `--export-pem` keep the classic formatters — explicitness is captured in `load_config` *before* the defaults merge, because `Config::default()` seeds `output` with `Some("summary")`. Host checks run on a bounded worker pool (`spawn_checks`, ≤16 threads) that **streams** `(index, HostOutcome)` over an mpsc channel; the CLI path drains it fully into input-order slots, the TUI consumes incrementally. `check_host` returns `Result<TLS, TLSError>` — error presentation belongs to the frontend (CLI logs, dashboard renders a failed row). `warning_label()` is the single source for `SecurityWarning` display labels across all three frontends. Hosts are parsed by `parse_host_port` (accepts `host`, `host:port`, and `https://host:port`; strips IPv6 brackets; rejects invalid ports). Exit-code logic: non-zero `--exit-code` is only applied when a cert is expired/revoked, below `--min-validity`, or (with `--fail-on-error`) a host could not be checked. `--scan`, `--export-pem`, `--ct-check`, `--fail-on-error`, and `--generate-config` are **pure CLI flags read directly off `Args`** — they are deliberately *not* part of `Config`/`FinalConfig` (so config-merge code and its many test literals stay untouched). `--export-pem` bypasses the formatters and prints `certificate.pem` per host.

**`src/tui/` — interactive dashboard (binary-only, `tui` feature).** `mod.rs` owns the event loop: `ratatui::init()` (installs a terminal-restoring panic hook), drain the results channel with `try_recv`, redraw when dirty, poll crossterm input at 100ms; keys are dispatched per `Screen` — Fleet: j/k/arrows move, g/G, ⏎ opens the explorer, q/Esc quit; Detail (full-screen certificate explorer with every field: subject/issuer DNs, validity, serial, fingerprints, SANs, chain, SCTs, grade reasons, scan results): j/k scroll, PgUp/PgDn/Space page, g/G, Esc/⏎/Backspace back, q quits. `state.rs` holds `App` (labels + `Vec<Option<HostOutcome>>` slots + selection + `Screen` + `detail_scroll`) and the `Verdict` logic — Critical: expired/revoked/≤15d; Warning: self-signed/any security warning/≤30d — which, unlike the old summary Status column, reflects *all* signals. `view.rs` is pure rendering (`draw(frame, &App)`): fleet list + tally on the left, detail pane (facts, expiry `LineGauge` fed by the `valid_*_unix` fields, per-category grade gauges, warnings) on the right; fully testable via `ratatui::backend::TestBackend` (see its tests). While the TUI is active, the `TUI_ACTIVE` atomic makes `tracing_writer` discard diagnostics so they can't corrupt the alternate screen. The dashboard forces `calculate_grade = true` (offline, cheap). Hosts still pending when the user quits are returned as `None` and deliberately **not** counted as errors.

**Config precedence (spread across `main.rs::load_config` and `config.rs::merge_with`):** defaults (`Config::default`) → config file (`--config <path>`, or `tlschecker.toml` in CWD if present) → CLI args (highest). Every field is `Option<T>` so merging is "last `Some` wins". `Config::example_toml()` is the canonical config schema (the README's inline TOML example mirrors it — keep them in sync).

**`src/metrics/prom.rs` — Prometheus push-gateway export** (`--prometheus`). Pushes per-host gauges (`tlschecker_days_before_expired`, `tlschecker_revocation_status` = 0/1/2/3, `tlschecker_grade_score`, `tlschecker_below_min_validity`) with host/cipher/issuer/grade labels. Gated behind the `--prometheus` flag.

## Conventions

- Errors use `thiserror` (`TLSError` in `lib.rs`, `ConfigError` in `config.rs`); the binary uses `anyhow`. Map new failure modes onto an existing `TLSError` variant where possible.
- Public items carry rustdoc; doc examples compile as part of `cargo test` (doc-tests), so keep `# Example` blocks valid.
- Weak-algorithm detection (`is_weak_algorithm`) matches on substrings/OIDs of the signature algorithm string — extend the match list there rather than introducing a new mechanism.
